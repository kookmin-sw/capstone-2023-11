인증과 인가의 개념

인증

  • 인증(authentication)은 사용자가 누구인지 확인하는 단계 → 로그인
    • DB에 저장된 ID/PW를 사용자가 입력한 값과 같은지 확인하는 단계
    • 로그인에 성공(인증)하면 서버는 응답으로 사용자에게 토큰(token)을 발급

인가

  • 인가(authorization)은 인증을 통해 검증된 사용자가 애플리케이션 내부의 리소스에 접근할때, 접근 권한이 있는지 확인하는 과정
    • 게시판에 접근 등급에 따라서 접근허가 여부를 결정하는 것이 하나의 예시가 됨
  • 사용자가 인증단계에서 받은 토큰(token)에는 이러한 인가 내용을 포함하고 있고 서버는 토큰을 통해 권한 유무 등을 확인하여 수행 함

스프링 시큐리티

  • 스프링 시큐리티에서는 인증,인가 등의 보안 기능을 제공하는 스프링 하위 프로젝트임

로그인/회원가입 플로우에 대한 반영사항

  • 이거 토큰 언제 던져주고 DB에 언제 가입 정보 넣어줄지 생각해야함
  • 근데 생각해보니까 토큰으로 DB에 저장하는게 아니라 받은 토큰을 통해서 서버에서 카카오 로그인 정보를 받아올 수 있기 때문에 여기서 1차적으로 토큰을 서버에 던져줘서 로그인 정보가 있는지 파악해야함
  • 그리고 회원정보가 없다면 회원가입 플로우로 이동시킴
  • 이때(회원가입 플로우로 이동시킬때) 서버에서 유저의 고유 UID값을 던져주도록하고 이 값을 가지고 1-2-1 노인 회원가입뷰로 이동시킨다
  • 2-1에서 회원가입 버튼을 누르면 회원의 기본 인적사항 + UID값을 요청으로 전송하고 이를 기반으로 DB에 최종 데이터를 쌓아 준다--- 수정 사항 ----그렇게 하면 UID값 이외의 프로필 이름이나 이름과 같은 항목을 최종 알수가 없을거 같다. --- 수정 사항 ----그렇게 하면 UID값 이외의 프로필 이름이나 이름과 같은 항목을 최종 회원가입 단계에서는 알수가 없을거 같다. 그렇기 때문에, 회원가입을 해야하는 상황이면 클라이언트가 서버에게 토큰을 전달하고 서버에서 토큰을 통해 인가하여 얻은 데이터를 다시 응답으로 클라이언트에 뿌려준다.
    • 그렇게 하면 UID값 이외의 프로필 이름이나 이름과 같은 항목을 최종 회원가입 단계에서는 알수가 없을거 같다.
    • 그렇기 때문에, 회원가입을 해야하는 상황이면 클라이언트가 서버에게 토큰을 전달하고 서버에서 토큰을 통해 인가하여 얻은 데이터를 다시 응답으로 클라이언트에 뿌려준다. 그렇게 하여 1-2-1의 뷰에서 이름(가칭, 김딸기)와 프로필 이미지 등등을 사용 할 수 있도록

수정사항 (3/17 (금)에 이야기해보기)

  • 우리 프로젝트에서 유저정보 관리시 “uid”으로 관리하고자 했지만, 레퍼런스를 찾아보니 이메일을 통해서 사용함
  • 로그인/회원가입 로직 결정사항 관련된 이야기 피그마에 코멘트 달아놓음( 최종 뷰)
